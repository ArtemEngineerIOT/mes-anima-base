# Регистрация событий

| Поле | Значение |
|------|----------|
| **Назначение** | Обеспечить оператору единый интерфейс для регистрации производственных событий с фиксацией параметров для последующей прослеживаемости и производственной аналитики. |
| **Основной актор** | Оператор |
| **Второстепенные акторы** | Начальник смены (контроль, коррекция), администратор (настройка справочников), система сбора данных с машин (PLC/датчики) |

## Предусловия

1. Оператор авторизован в MES на рабочем месте, привязанном к конкретной машине.
2. На машине активен производственный этап (или разрешена работа вне этапа для перемотки).
3. В системе настроены справочники: коды причин с пометкой необходимости указания времени / метража / комментария; иерархия «зеропулов» (попыток настроиться); список линий (PM / Passer продукта по спецификации).
4. С датчиков машины поступают данные о скорости, метраже, ударах ножа и т. д.

## Основной поток событий

1. Система фиксирует факт события (автоматически или вручную). Для пользователя отображается оповещение (тревога), а также графическое отображение в блоке «Регистрация события».
2. Оператор разворачивает блок «Регистрация события» (пошаговая форма).
3. **Шаг 1 — определение кода события:**
   - оператор выбирает код причины из справочника; по признакам необходимости времени / метража / комментария обозначаются обязательные и необязательные поля;
   - указывает, удалять ли брак, вызванный событием, сразу: по умолчанию при остановке машины — «Да», при регистрации на ходу (без сигнала остановки) — «Нет»;
   - нажимает «Далее».
4. **Необработанные события.** Если в системе были зарегистрированы, но по каким-либо причинам остались необработанными события, они отображаются таблицей с возможностью выбора события для обработки и удаления с комментарием.
5. **Шаг 2 — детали события:**

   **Удаляем брак:**
   - поля «Метраж от / до» (с датчика, с возможностью корректировки вручную);
   - поля «Время начала / окончания события» (с датчика, с возможностью корректировки вручную);
   - галочка «Весь этап» для событий, длящихся всё время исполнения этапа;
   - привязка к рулону на намотке (по умолчанию 0 — рулон для брака);
   - комментарий;
   - для настройки (код 120): детализация через подкоды (зеропулы) — оператор может указать причины дополнительных заездов (буквы A, B, C…).

   **Брак регистрируем (будет удалён на следующей стадии):**
   - привязка к рулону на намотке (по умолчанию — активный рулон на намотке);
   - выбор линий: сторона (PM / Passer), номера линий (один, несколько через запятую, диапазон или «все линии»); количество линий подтягивается из информации по текущему заказу;
   - поля «Метраж от / до» (с датчика, с возможностью корректировки вручную);
   - карточки начала и конца;
   - поля «Время начала / окончания» (когда было замечено начало и конец брака);
   - комментарий.

   Обязательность полей берётся из справочника кодов (колонки: время, метраж, комментарий).

6. **Шаг 3 — подтверждение и регистрация:**
   - система показывает сводку введённых данных;
   - оператор нажимает «Зарегистрировать»;
   - запись сохраняется в журнале процесса этапа в полном объёме указанных данных.
7. **Журнал процесса:**
   - журнал обновляется динамически; каждое зарегистрированное событие сразу отображается в журнале;
   - для оператора журнал отображается без возможности изменения данных;
   - для начальника смены, его помощников и администраторов журнал отображается в виде редактируемых данных;
   - оператор не может просмотреть журнал процесса по произвольному этапу — только по активному этапу на машине;
   - начальник смены и администратор могут просматривать и редактировать любой выбранный журнал, в том числе по закрытым этапам.
8. JSON-пакет в 1С (или таблица — уточнить на стадии проектирования).

## Альтернативные потоки

### А1: Остановка с одновременным возникновением нескольких причин (параллельные события)

- В журнале процесса допускается регистрация нескольких событий, перекрывающихся по времени (например, простой «Устранение розлива» и внутри него брак «Тонирование»).
- Для корректного расчёта OEE система использует принцип одного активного кода в момент времени (суммарное время событий не должно превышать время этапа), но аналитика брака и простоев ведётся раздельно с возможностью пересечения. При необходимости расчёта логики OEE можно намеренно исключать вложенность при подготовке данных для отчётов; хранить данные нужно в полном корректном объёме.

### А2: Данные поступают очень часто («троит») (US-139)

1. В интерфейсе администратора предусмотрена настройка тайм-аута для объединения повторяющихся сигналов в группу.
2. Оператор регистрирует группу повторяющихся сигналов, а не каждый по отдельности. В таблице «Необработанные события» такая группа выглядит как единая запись.
3. При необходимости администратор может запросить «сырые» данные — увидеть состав группы сигналов.

## Постусловия

1. В MES создана запись о событии с полными атрибутами.
2. Данные события доступны для отчётов.
